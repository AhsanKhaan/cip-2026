Critical Finding: Upstash + BullMQ Incompatibility BullMQ uses Redis BLPOP/BRPOP blocking commands and Lua scripts that require persistent TCP connections. Upstash's HTTP-based API does NOT support this. Using Upstash Pay-As-You-Go for BullMQ will either fail or generate thousands of extra commands per minute — costing you money every day even with zero users. Solution shown in diagram below: use TWO separate Redis instances.

🚨
BullMQ is NOT compatible with Upstash Pay-As-You-Go plan BullMQ requires persistent TCP Redis connections and BLPOP/BRPOP blocking commands. Upstash HTTP API does not support these. Even if TCP works via Upstash Fixed plan, BullMQ polls Redis aggressively every few milliseconds — generating 100,000s of commands per day, making Pay-As-You-Go extremely expensive. The official Upstash docs explicitly recommend switching to a Fixed plan for BullMQ workloads.
✅
Recommended Solution: Two Redis Instances Use Upstash (HTTP) for caching from Vercel — free tier is perfect. Use Local Redis on the same Hetzner VPS for BullMQ — costs $0 extra since it runs on your already-paid $4.51/mo server. This is the cleanest, cheapest architecture.

❌ AVOID — BullMQ

Upstash Pay-As-You-Go for BullMQ

BullMQ polls Redis every ~50ms even when idle.
That's ~1,728,000 Redis commands/day just from polling.
At $0.20 per 100K commands that's $3.46/day = $103/month just in Redis commands with zero users.

Additionally, Upstash HTTP doesn't support BLPOP/BRPOP. BullMQ will malfunction unpredictably.

❌ OVERKILL at launch

Upstash Fixed Plan ($10/mo)

Upstash's own docs recommend this for BullMQ workloads.
It does support TCP connections and handles BullMQ.

But why pay $10/mo for something you get for $0 using local Redis on your already-running Hetzner server?

Consider this only if Hetzner VPS is removed from your stack.

Architecture is Feasible — with one critical fix The overall design is solid and production-ready. One change required: never use Upstash for BullMQ. Use local Redis on Hetzner for BullMQ and keep Upstash only for HTTP cache/rate-limit from Vercel.
✅ CONFIRMED FEASIBLE

MongoDB Atlas M0 → Native Driver

Time-series collections for candles are the right choice. Native driver (no Mongoose) is mandatory for this. M0 free tier handles MVP easily. Upgrade to M10 at 50K users.

✅ CONFIRMED FEASIBLE

Upstash Redis for Cache/Rate-limit

500K free commands/month is more than enough for caching 4 metal prices and rate-limiting API routes. HTTP API works perfectly with Vercel serverless. Cost: ~$1/mo at launch.

✅ CONFIRMED FEASIBLE

BullMQ + Local Redis on Hetzner

Local Redis (apt install redis-server) on the same $4.51/mo Hetzner VPS. Zero extra cost. Full TCP compatibility with BullMQ. AOF persistence for job durability. This is the production-safe setup.

✅ CONFIRMED FEASIBLE

Binance WebSocket for Crypto

Persistent WS connection on Hetzner VPS is exactly right. Auto-reconnect with exponential backoff handles drops. Free forever, real-time BTC/ETH data. Never call Binance from Vercel — too slow and stateless.

✅ CONFIRMED FEASIBLE

Vercel + Hetzner Split

Vercel for Next.js (SSR/ISR/API) and Hetzner for persistent worker is the correct architectural split. Vercel's 10s timeout makes long-running workers impossible there. Hetzner at $4.51/mo is the right price point.

⚠ ONE CHANGE REQUIRED

Never: Upstash for BullMQ

This is the only critical issue in the current plan. Pay-As-You-Go Upstash + BullMQ = $100+/mo with zero users due to aggressive polling. Fixed plan ($10/mo) works but is wasteful when local Redis is free on your existing VPS.
